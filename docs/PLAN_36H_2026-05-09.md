## How autonomy actually fires

The mechanism is one of:

**Option A — `/loop` dynamic mode (preferred)**
At 22:00 CET, user runs `/loop` with a task description. I use
`ScheduleWakeup` at the end of each turn to fire the next iteration.
Cadence: 30–60 min when actively shipping; 90–120 min when waiting on
the harness; 270s only when polling a build/deploy.

**Option B — `cron` scheduled tasks**
At 22:00 CET, user creates a cron task that fires me every 60 min with
the autonomous-loop sentinel. I wake, read this plan, do one cycle,
sleep until the next cron tick.

Either works. **Option A is cheaper on context** because I can vary the
delay; option B is more predictable but burns one cache miss per fire.

If the user just kicks off `/loop` with the prompt **"continue the
36-hour plan in docs/PLAN_36H_2026-05-09.md"** I'll read this file and
self-pace.

1. **Read this doc** to find the active block.
2. **`tasklist | grep python`** — verify harness alive. If dead, relaunch
   via the playbook in `docs/AUTONOMOUS_WEEKEND_STATE.md`.
3. **Glance at newest hourly summary** in
   `data/explore_runs/20260509_062947_72h/hourly_*.md` —
   look at `Top console errors` + `Top network failures (status >= 400)`
   delta from last cycle. If a NEW error count is in the double digits,
   bump it to the front of the queue.
4. **Do one cycle of the active block** (target: 1–3 commits, 30–60 min).
5. **Build → commit → push → deploy** in that order. Never amend.
6. **Update this doc's "Cycle log" table** below.
7. **Schedule next wake-up** with a delay that matches actual progress:
   - mid-task on a multi-commit block → 1800s (30 min) so the harness
     has time to detect new bugs from the prior deploy
   - block boundary → 3600s (1 hour) for a clean cache miss
   - waiting on a long deploy → 270s, no longer (don't pay cache-miss
     for a 5-minute wait)

**Hard rule:** if the block is done and the next clock-time isn't here
yet, schedule wake-up to land at the next block's start, not earlier.
Don't burn quota grinding ahead.

## Context management

Each wake-up costs context. Strategy:

- **Short, focused wakes**: 1–3 commits per fire. Don't try to do
  the whole block in one cycle.
- **Persist findings to disk** via this doc's cycle log so I don't
  carry them in working memory.
- **Re-read this doc + AUTONOMOUS_WEEKEND_STATE.md** at the start of
  every cycle (small cost, high value).
- **Avoid Read-ing huge files** unless needed. Prefer Grep with
  narrow queries.
- **Don't re-verify what was just shipped** — Edit/Write/Bash report
  errors; trust the gate.

If context budget for a single wake hits ~70%, ship what I have and
sleep. Don't push to 90% — leaves no room for the next cycle's chain.

clean partial**. Then:

- If 1 block behind → trim block 6 (no handoff doc, just a 5-line
  summary).

  isolatable, lowest user-visible win).
- If 3+ blocks behind → drop blocks 3 and 5 polish items, keep i18n
  blocks (1 + 2) which are the most user-visible deliverables.

**Never ship broken.** A clean partial > a complete failure.
